*Presenter-facing. 5–7 beats, ~5 minutes. Everything here runs on the Mac (Node 3) — no Pis required. Rehearse the full sequence once before judges arrive.*

## The narrative

**A destroyer is a self-contained city.** Machinery keeps the lights on. Propulsion drives the shaft. Navigation holds the track. Sonar hunts below the surface. C2 holds the link. The auxiliaries balance the grid. Own-systems watches the hull and UUV. Contacts tracks what's out there. Each of these is a distinct system, on a distinct bus, run by a distinct crew.

Theseus mirrors that architecture: **each subsystem runs its own onboard ML model in its own container**, isolated, low-RAM, inference-only at the edge. A human watch officer is always in command — the system recommends, the officer decides, and every decision is sealed in a tamper-evident record.

When the ship is **not in a contested environment**, it syncs **model deltas — never raw sensor data** — to the fleet brain ashore (Node 3 MLflow in the demo). The fleet brain merges only the **provenance-attested** updates. A captured node can't poison the fleet: its delta is rejected at the gate because it lacks the signed attestation chain. A frozen model can't hide: every update is eval-gated before it promotes. The result is a fleet that keeps learning at sea — **Tesla-fleet-style, but human-authorized, eval-gated, and provenance-attested**. That sealed record is not a side feature: it is what makes fleet learning **accreditable** — a cATO-for-AI path, in the AO's own OSCAL language.

The demo runs **three live destroyers** — USS Theseus (DDG-118), USS Daedalus (DDG-119), USS Ariadne (DDG-120) — with **21 live containers total**: 18 subsystem containers (6 per hull × 3 hulls; the flagship DDG-118 runs 8/8 subsystems live including the AE own-systems node), 2 UUV Pi-emulation nodes (sonar/contacts, :54321/:54322), and 1 AE own-systems container (`uuv-ae:latest`, :54547). All stream simultaneously with per-hull phase offsets so they don't move in lockstep.

> **What's live vs. sealed (say this if asked):** the **per-subsystem anomaly scores** update in real time from the containers — that's the live edge inference. The per-hull **"LOCAL MODEL" / "SIGNED Δ → SHORE"** stats are the **last sealed fleet-merge** from the tamper-evident record (regenerated by `deploy/demo_up.sh`), not a per-second ticker — fleet-learning syncs happen when a ship reconnects, not continuously. Both are real; only the subsystem scores change live.

## Primary run path — one command

```bash
# Bring up the full strike group: MLflow + 9 registered models + 2 UUV nodes
# + 3 destroyers (21 containers: 18 subsystem + 2 UUV nodes + 1 AE own-systems), all live-fed, API on :8501.
bash deploy/strike_group_up.sh

# Then in a second terminal:
cd frontend/ui && npm run dev
# Open http://localhost:5173 → select "STRIKE GROUP"

# Tear down:
bash deploy/strike_group_down.sh
```

`strike_group_up.sh` handles the full sequence:
1. Node-3 MLflow `:5050` (starts if not running)
2. Register all 9 subsystem + AE models `@production` (skip with `--fast` if already registered)
3. Two UUV Pi-emulation nodes (`deploy/pi-emulation/up.sh --feed`) — :54321/:54322
4. Three destroyer hulls via `deploy/ship-emulation/up.sh --fleet --feed` (18 subsystem containers, per-hull phase/seed offset) + 1 AE own-systems container (`uuv-ae:latest`, :54547); flagship DDG-118 runs 8/8 live
5. Demo API on `:8501` (`/api/destroyer`, `/api/state`, `/api/mlflow`, `/api/fleet`)

The demo also includes a second launch script for the tamper-evident records and fleet flywheel:

```bash
bash deploy/demo_up.sh   # pre-populates tamper-evident records + flywheel + preflight checks
```

Run both before a high-stakes demo.

## If asked — cATO framing

The cATO claim is a **mechanism, not a delivered fact.** If a judge asks whether you have an ATO:

> *"We emit OSCAL assessment-results and in-toto/DSSE attestations so an Authorizing Official can accredit the learning pipeline, not frozen weights. No AO has signed this record yet — the mechanism is what's novel, and it's demonstrable right now."*

The value is that the *process* of updating the model is accreditable. Frozen-weight ATOs break every time the model improves. Ours doesn't.

## If asked — Byzantine robustness

The provenance gate alone is necessary but not sufficient. Three layers work together:

1. **Provenance gate** — rejects unsigned or forged deltas (unknown `keyid` not in the trust registry).
2. **Eval gate** — rejects a merged model that regresses on held-out evaluation data, *even if all contributing deltas were properly attested*. A captured node with a valid key that submits a subtly bad model fails here.
3. **Rollback** — falls back to the last-good registered model automatically.

If asked whether this is Byzantine-robust: *"The eval-gate is the robust-aggregation backstop. Provenance alone — which is what most FL proposals stop at — is necessary but not sufficient. We require the merged model to pass an eval gate before it promotes, regardless of how clean the attestation chain is."*
